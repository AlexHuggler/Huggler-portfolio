## Problem

Fraud teams in telecom and fintech need to react in seconds, not hours. Batch
ETL surfaces compromised accounts long after the damage is done. A streaming
fraud-signals pipeline ingests transaction events, applies rules and statistical
anomaly detection, and writes scored events to a lakehouse where analysts can
slice further with dbt. This project demonstrates the production patterns that
matter: exactly-once semantics, watermark-based late-event handling, dbt tests
on the resulting marts, and a lightweight dashboard for operators.

## Stack

- **Kafka** for the event bus (replayable, ordered per partition, decouples producer from consumer).
- **Spark Structured Streaming** for the consumer (mature exactly-once with checkpoints, broad ecosystem support).
- **Delta Lake** for ACID lakehouse storage (cheap object store, fast incremental reads, built-in MERGE).
- **dbt** for the analytics layer (testable, reviewable SQL with explicit lineage).
- **Streamlit** for the operator UI (fast to ship, runs from a single file, no JS build step).

## What I would do differently in production

- Move anomaly detection to a dedicated feature store with versioned features.
- Replace per-event Z-score with online algorithms (Welford or t-digest) to bound memory.
- Add a schema registry (Confluent or AWS Glue) so producer/consumer compatibility is enforced at deploy time.
- Push the dashboard onto a lightweight web framework with role-based access (Streamlit is fine for ops, not for customer-facing).

See the repo for full setup, tradeoffs, and limitations.
